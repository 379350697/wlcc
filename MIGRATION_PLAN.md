# Claude Code 研究结论落地迁移方案

## 1. 文档目的

本文档定义本轮改造的迁移策略，确保在引入 Skill、任务状态、恢复机制和项目事实源规范时：
- 不破坏现有可用能力
- 不丢失历史数据
- 不出现一次性硬切换
- 出现问题时可快速回退

本方案遵循原则：
**先兼容、再迁移、后切换。**

---

## 2. 迁移对象

本轮迁移涉及 4 类对象：

1. 项目事实信息
2. 任务状态信息
3. 会话恢复信息
4. 执行留痕信息

对应新增或规范化的载体包括：
- `README.md`
- `STATUS.md`
- `DECISIONS.md`
- `TASKS.md`
- `INCIDENTS.md`
- Resume State 文件
- 变更/失败/回滚日志

---

## 3. 迁移原则

### 原则 1：旧结构先不删
迁移初期保留旧读取路径，不删除原有文件或历史记录。

### 原则 2：新结构先用于新增内容
先让新任务、新摘要、新状态进入新结构，再逐步吸收旧内容。

### 原则 3：优先迁移“当前活跃信息”
优先处理当前正在用、后续要继续执行的信息，不追求一次迁完所有历史。

### 原则 4：迁移过程要可验证
每一步都要有对照输入与输出，证明迁移后信息不缺关键信息。

### 原则 5：失败时优先保留旧能力
当新结构不可靠时，优先降级到旧结构读取，而不是强行坚持新结构。

---

## 4. 迁移范围与非范围

### 4.1 纳入迁移范围
- 项目级状态文档规范化
- 当前任务状态显式化
- Resume State 引入
- Skill 对项目事实文件的优先读取
- 安全写入与执行留痕

### 4.2 不纳入本轮迁移范围
- 全局长期记忆底层重构
- OpenClaw 内核级 memory 插件替换
- 重型数据库化任务系统
- 多 agent runtime 改写
- UI / bridge / remote 控制改造

原因：
这些方向超出本轮“最小闭环、可回滚”的目标范围。

---

## 5. 迁移阶段

## Phase 0：基线确认

### 目标
在任何改动前，确认当前状态和基线文件。

### 需要确认的内容
- 当前项目目录结构
- 现有文档是否存在：`README.md` / `STATUS.md` / `DECISIONS.md` / `TASKS.md` / `INCIDENTS.md`
- 当前是否已有任务清单或执行记录
- 当前是否已有手工总结、研究结论、执行计划

### 输出
- 基线文件清单
- 缺失文件清单
- 当前活跃任务清单

### 验证标准
- 能明确知道哪些信息已有，哪些需要初始化

---

## Phase 1：Skill 层先行

### 目标
在不动底层结构的前提下，先通过 Skill 建立统一输入输出。

### 实施内容
1. 实现 `task-extract`
2. 实现 `project-state`
3. 实现 `context-compact`
4. 实现 `handoff-report`

### 迁移策略
- 先只生成或更新项目文档
- 不要求底层状态结构已完全成型
- 通过 Skill 输出反向逼出最小文档规范

### 成功标准
- 能从真实聊天中更新 `TASKS.md`
- 能从项目目录生成 `STATUS` 类摘要
- 能生成结构化压缩摘要

### 风险控制
- 所有 Skill 初期只允许写低风险文件
- 写入前先读取旧文件并保留原结构

---

## Phase 2：项目事实源规范化

### 目标
把项目事实从“散落文本”变为统一入口文件。

### 目标结构
```text
projects/<project>/
├── README.md
├── STATUS.md
├── DECISIONS.md
├── TASKS.md
└── INCIDENTS.md
```

### 迁移策略
- 已存在的文件继续沿用
- 缺失的文件按最小模板补齐
- 不要求一次性补全全部历史内容
- 优先填充当前任务需要的信息

### 数据来源优先级
1. 已存在项目文档
2. 已确认的研究/计划文档
3. 当前会话中的明确结论
4. 旧聊天摘要（仅作补充）

### 验证标准
- 项目核心信息可通过固定文件入口读取
- 缺失信息有明确标记，不假装完整

---

## Phase 3：任务状态迁移

### 目标
让任务状态从“聊天隐式存在”迁移到“显式状态存在”。

### 迁移对象
- 当前进行中任务
- 当前 blocked 任务
- 最近完成但仍需追踪结果的任务

### 不优先迁移对象
- 已长期归档且不再恢复的历史任务
- 仅闲聊中出现、无明确执行意义的事项

### 迁移步骤
1. 从现有聊天/任务单中抽取活跃任务
2. 生成统一任务记录
3. 为每个 doing / blocked 任务生成 Resume State
4. 校对任务字段是否完整

### 最小字段要求
- 目标
- 当前状态
- 最近结果
- 阻塞项
- 下一步
- 更新时间

### 验证标准
- 不看完整聊天历史，也能理解任务当前状态
- 任务可区分 todo / doing / blocked / done

---

## Phase 4：恢复摘要迁移

### 目标
为活跃任务建立 Resume State，降低恢复成本。

### 迁移策略
- 只给活跃任务生成
- 优先生成 doing / blocked 任务的 Resume State
- done 任务不强制补 Resume State，除非后续仍需追踪

### 结构模板
```md
## Resume State
- 最后目标：
- 最后成功动作：
- 最后失败动作：
- 当前阻塞：
- 建议下一步：
```

### 验证标准
- 任一活跃任务可在不重读全部历史的前提下恢复 80% 以上执行上下文

---

## Phase 5：安全写入与留痕接入

### 目标
避免状态迁移过程本身造成状态污染。

### 接入顺序
1. `TASKS.md`
2. `STATUS.md`
3. Resume State
4. 其他新增状态文件

### 写入策略
- 先读旧内容
- 生成新内容
- 写入临时文件
- 校验结构
- 原子替换
- 成功/失败写入日志

### 验证标准
- 模拟失败时原文件仍完整
- 日志可看到失败原因和影响对象

---

## 6. 兼容读取策略

迁移期内采用“新优先，旧兜底”的读取策略。

### 读取顺序
1. 新项目事实文件
2. 新任务状态文件 / TASKS
3. Resume State
4. 旧文档/旧摘要
5. 聊天历史

### 降级规则
- 若新文件缺失，明确标注并读旧结构
- 若新文件损坏，停止写入并切回旧读法
- 若新旧冲突，以项目事实文件和最近明确状态为准，并标记冲突待人工确认

---

## 7. 数据迁移策略

### 7.1 项目事实类
迁移方式：人工校对 + Skill 辅助生成

原因：
- 这类内容事实性高
- 错误写入成本高
- 不适合盲目自动归并

### 7.2 任务状态类
迁移方式：Skill 抽取 + 结构化审校

原因：
- 任务信息格式较稳定
- 适合先自动提取，再人工验收

### 7.3 恢复摘要类
迁移方式：按需生成，不回补全部历史

原因：
- 恢复摘要是执行态信息
- 只对活跃任务有价值

### 7.4 留痕类
迁移方式：从新版本开始记录，不追补旧全量历史

原因：
- 审计日志追补意义有限
- 成本高、收益低

---

## 8. 风险点与应对

### 风险 1：旧信息缺失，导致新结构内容不完整
应对：
- 明确标记“未发现/待验证”
- 不用模型推测补空

### 风险 2：Skill 抽取结果过度主观
应对：
- 固定输出结构
- 区分事实/推测/待确认
- 初期采用人工复核

### 风险 3：迁移后新旧状态冲突
应对：
- 以项目事实文件优先
- 冲突项单独列出待确认
- 暂不自动覆盖

### 风险 4：写入失败污染文件
应对：
- 使用临时文件 + 原子替换
- 替换失败立即停止写入

### 风险 5：迁移面过大导致实施失控
应对：
- 只迁当前活跃项目与活跃任务
- 历史长尾延后处理

---

## 9. 验证计划

### 9.1 Skill 验证
至少使用 3-5 组真实任务对话，验证：
- task-extract 是否能稳定抽取任务单
- project-state 是否能在缺文件时降级
- context-compact 是否能区分已证实/待验证
- handoff-report 是否能生成两种稳定视图

### 9.2 状态迁移验证
验证项：
- doing 任务是否都有 nextStep
- blocked 任务是否都有 blocker
- Resume State 是否可读可恢复

### 9.3 安全写入验证
验证项：
- 模拟写入失败后原文件不损坏
- 重复写入不会产生结构错乱

### 9.4 兼容验证
验证项：
- 新文件缺失时系统可读旧结构
- 新旧并存时优先级行为一致

---

## 10. 切换条件

只有同时满足以下条件，才允许把新结构作为默认主路径：

1. Skill 层已经稳定输出
2. 项目事实文件入口已建立
3. 当前活跃任务已迁移到显式状态
4. Resume State 已对活跃任务生效
5. 安全写入和留痕已验证通过
6. 新旧兼容读取验证通过

未满足前，仍处于迁移期，不进行硬切换。

---

## 11. 最终结论

本轮迁移不追求一次性“重构所有记忆与状态系统”，而是分阶段建立：
- 稳定的项目事实入口
- 显式任务状态
- 可恢复的 Resume State
- 可追踪的写入与留痕

迁移策略必须是：
**旧结构不断、活跃信息先迁、新结构先验证、验证通过再切换。**
