# Claude Code 研究结论落地架构方案

## 1. 文档目的

本文档用于把当前研究结论转换为可执行、可验证、可回滚的实施架构。

约束如下：
- 先做最小可用闭环，不做重型平台化
- 能通过 Skill 达到 80% 效果的能力，优先 Skill 化
- 只有需要强一致性、恢复能力、审计能力的部分才下沉到底层
- 所有底层改动必须具备兼容层、迁移方案、回滚方案
- 不把研究报告中的未验证细节直接当作实现事实

---

## 2. 设计目标

本轮改造目标不是复刻某个 Claude Code 项目，而是解决当前 Agent 体系中最影响稳定性的 5 个问题：

1. 任务要求散落在聊天历史中，难以稳定提取和恢复
2. 项目事实、用户偏好、临时摘要、任务状态混在一起，易污染
3. 长对话上下文膨胀后，恢复依赖人工重读历史
4. 风险操作缺少统一分级与执行留痕
5. 底层结构一旦改动，缺少清晰的兼容和回滚路径

因此，本方案追求的不是“功能总量最大化”，而是：
- 更稳
- 更轻
- 更可恢复
- 更可验证
- 更可逐步演进

---

## 3. 第一性原理与架构判断

### 3.1 为什么先做状态与分层，而不是先做更多能力

Agent 系统的核心约束来自四个事实：

1. 上下文窗口有限
2. 外部动作存在失败与中断
3. 写操作天然带风险
4. 复杂任务天然跨多步、跨时间

因此，系统必须优先解决：
- 哪些信息长期保留
- 哪些信息作为项目事实源
- 哪些信息属于当前任务状态
- 任务中断后如何恢复
- 风险动作如何拦截、确认、留痕

结论：
**本轮优先建设“状态、事实、恢复、留痕”四类结构，不优先建设重型交互层、多角色层、复杂桥接层。**

---

## 4. 总体架构

本轮采用“两层架构”：

### Layer A：流程能力层（Skill）
用于快速落地高价值、低风险、可迭代能力。

计划落地的 Skill：
1. `task-extract`
2. `project-state`
3. `context-compact`
4. `handoff-report`

### Layer B：运行时支撑层（底层能力）
用于承接必须具备一致性与恢复性的能力。

计划建设的底层能力：
1. 项目事实源规范
2. 任务状态底座
3. Resume State 恢复机制
4. 安全写入与执行留痕
5. 最小风险分级
6. 记忆分层兼容读取策略

### 设计原则
- Layer A 先上线，拿到可用结果
- Layer B 只实现 Skill 无法等价替代的部分
- Layer B 先做兼容层，再做切换

---

## 5. 信息分层架构

### 5.1 分层目标

把以下几类信息拆开管理：
- 长期规则/人格/稳定偏好
- 项目事实
- 当前任务状态
- 会话压缩摘要
- 执行留痕/恢复状态

### 5.2 分层结构

建议采用以下分层：

#### A. 长期规则层
职责：
- 风格
- 工作纪律
- 稳定偏好
- 禁止事项

来源示例：
- `SOUL.md`
- `AGENTS.md`
- `USER.md`
- `MEMORY.md`

特点：
- 更新频率低
- 应避免被项目噪音污染

#### B. 项目事实层
职责：
- 项目定义
- 关键约束
- 当前阶段
- 决策记录
- 事故记录

建议位置：
- `projects/<project>/README.md`
- `projects/<project>/STATUS.md`
- `projects/<project>/DECISIONS.md`
- `projects/<project>/TASKS.md`
- `projects/<project>/INCIDENTS.md`

特点：
- 事实性高于聊天上下文
- 应作为项目判断的首要依据

#### C. 任务运行层
职责：
- 当前任务目标
- 状态流转
- 最近结果
- 阻塞项
- 下一步
- 恢复状态

建议位置：
- `projects/<project>/.agent/tasks/`
- `projects/<project>/.agent/resume/`
- 或先以 `TASKS.md` + `RESUME.md` 最小实现

特点：
- 生命周期与任务绑定
- 应显式恢复
- 不应混入长期偏好

#### D. 会话压缩层
职责：
- 长对话摘要
- 工具结果摘要
- 临时上下文压缩

建议位置：
- `projects/<project>/.agent/context/`
- 或仅作为 Skill 输出结果先行存在

特点：
- 可替换、可压缩
- 不承担长期事实职责

#### E. 审计与恢复层
职责：
- 执行留痕
- 写入结果
- 失败记录
- 回滚说明

建议位置：
- `projects/<project>/.agent/logs/`
- `projects/<project>/.agent/audit/`

特点：
- 面向恢复与排查
- 不参与事实推理优先级竞争

---

## 6. 事实源优先级

为避免“聊天里一句临时话覆盖项目事实”，定义统一读取优先级：

1. 项目事实文件（README / STATUS / DECISIONS / INCIDENTS）
2. 当前任务状态文件
3. Resume State
4. 当前会话内的最新明确指令
5. 会话压缩摘要
6. 一般聊天历史
7. 模型内部生成的推断

规则：
- 摘要不能覆盖事实源
- 推断不能覆盖已记录事实
- 缺失信息必须显式标记“未发现/待验证”

---

## 7. Skill 层架构设计

### 7.1 task-extract

#### 目标
把聊天内容整理为结构化任务单，并更新项目任务状态。

#### 输入源优先级
1. 用户当前明确需求
2. 项目现有 `TASKS.md`
3. 项目现有 `STATUS.md`
4. 当前会话已证实执行结果

#### 输出结构
```md
## Task
- 目标：
- 当前状态：
- 阻塞项：
- 下一步：
- 验收标准：
```

#### 行为约束
- 优先追加/合并，不粗暴重写全部任务内容
- 对不确定项标记“待确认”
- 不擅自关闭未完成任务

---

### 7.2 project-state

#### 目标
读取项目核心文件，生成当前状态摘要。

#### 输入文件
- `README.md`
- `STATUS.md`
- `DECISIONS.md`
- `TASKS.md`
- `INCIDENTS.md`

#### 输出结构
```md
## Project State
- 当前阶段：
- 最近变化：
- 风险点：
- 阻塞项：
- 下一步：
```

#### 行为约束
- 缺文件时必须标明“缺失/未发现”
- 事实优先于聊天历史
- 不把旧状态当最新状态，必要时标注时间不明

---

### 7.3 context-compact

#### 目标
把长对话压缩为可恢复执行的结构化摘要。

#### 输出结构
```md
## Compact Summary
### 已证实
- 

### 已决策
- 

### 待验证
- 

### 当前任务状态
- 

### 下一步
- 
```

#### 行为约束
- 必须区分事实与推测
- 不输出流水账式聊天复述
- 关键文件、阻塞项、下一步不得丢失

---

### 7.4 handoff-report

#### 目标
把结果转换为 CEO/Coder 可直接消费的交付格式。

#### CEO 输出
- 结论
- 风险
- 待确认项

#### Coder 输出
- 目标
- 范围
- 文件
- 验收标准

#### 行为约束
- 同一输入应可稳定生成两种视图
- Coder 版必须具体到执行，不允许空话

---

## 8. 底层能力架构设计

### 8.1 项目事实源规范

#### 目标
建立统一项目状态载体，降低聊天依赖。

#### 最小目录规范
```text
projects/<project>/
├── README.md
├── STATUS.md
├── DECISIONS.md
├── TASKS.md
└── INCIDENTS.md
```

#### 作用
- `README.md`：项目定义与边界
- `STATUS.md`：当前阶段、最近变化、阻塞、下一步
- `DECISIONS.md`：关键决策及理由
- `TASKS.md`：任务列表与当前状态
- `INCIDENTS.md`：异常、故障、恢复经验

---

### 8.2 任务状态底座

#### 目标
让任务状态显式化、可恢复、可追踪。

#### 最小状态模型
```ts
Task {
  id: string
  project: string
  goal: string
  status: 'todo' | 'doing' | 'blocked' | 'done'
  latestResult: string
  blocker: string
  nextStep: string
  updatedAt: string
}
```

#### 最小能力
- 创建任务
- 更新状态
- 标记阻塞
- 记录最近结果
- 恢复当前上下文

#### 实现建议
- 第一阶段可采用 Markdown + 结构化段落
- 第二阶段再视需要下沉到 JSON/YAML 状态文件

---

### 8.3 Resume State

#### 目标
任务中断后快速恢复，不依赖重读全量历史。

#### 最小结构
```md
## Resume State
- 最后目标：
- 最后成功动作：
- 最后失败动作：
- 当前阻塞：
- 建议下一步：
```

#### 更新时机
- 任务进入 doing
- 任务关键步骤完成后
- 出现失败或阻塞后
- 任务交接前

---

### 8.4 安全写入

#### 目标
避免写入中断污染原状态。

#### 强制策略
- 先写临时文件
- 内容校验通过后原子替换
- 替换失败保留原文件
- 失败写入记录日志

适用范围：
- `TASKS.md`
- `STATUS.md`
- `DECISIONS.md`
- Resume State
- 底层状态文件

---

### 8.5 执行留痕

#### 目标
提供恢复、审计、排查能力。

#### 最低记录字段
- 时间
- 操作模块
- 操作对象
- 动作类型
- 成功/失败
- 结果摘要
- 回滚建议（如适用）

#### 建议分类
- 变更日志
- 失败日志
- 回滚说明

---

### 8.6 风险分级

#### 目标
统一执行前风险判断口径。

#### 最小分级
- L0：只读分析
- L1：低风险写入（文档、报告、说明）
- L2：中风险变更（脚本、配置、构建）
- L3：高风险变更（删除、重启、运行态系统、生产环境）

#### 规则
- 默认保守
- L0 默认可执行
- L1 可执行但需留痕
- L2 建议明确确认
- L3 必须逐次确认，并附回滚说明

---

## 9. 兼容层设计

### 9.1 总原则
不一次性替换旧结构，先做兼容读。

### 9.2 兼容策略
- 旧结构继续可读
- 新结构优先用于新增写入
- 在验证期内允许双读
- 切换前保留回退路径

### 9.3 兼容期行为
- 若新状态不存在，降级读旧状态
- 若项目事实文件缺失，Skill 输出时明确标记
- 若 Resume State 缺失，退化为读取 TASKS + STATUS

---

## 10. 分阶段实施方案

### Phase 1：Skill 快速落地
交付：
- `task-extract`
- `project-state`
- `context-compact`
- `handoff-report`

目标：
- 先解决任务漂移、长对话难恢复、交付口径不统一

### Phase 2：状态与恢复底座
交付：
- 项目事实源规范
- 任务状态底座
- Resume State
- 安全写入与留痕

目标：
- 解决任务中断不可恢复、项目状态依赖聊天、状态写入不安全

### Phase 3：风险与更深层治理
交付：
- 最小风险分级
- 更强兼容读策略
- 进一步自动化治理

目标：
- 提升可控性与长期稳定性

---

## 11. 不纳入本轮的内容

以下方向本轮不作为优先实施项：

1. 重型多 agent/team 运行时
2. 复杂 UI / bridge / remote transport
3. 全自动长期记忆大系统
4. 大而全 hook 平台
5. 未验证细节驱动的底层重构

原因：
- 当前主矛盾不在入口和角色数量，而在状态、恢复、事实源、可追踪性

---

## 12. 验收标准

### 12.1 Skill 层验收
- 能从真实聊天中稳定提取任务单
- 能在项目文件不完整时稳定降级输出状态摘要
- 能把长对话压缩为结构化恢复摘要
- 能输出 CEO/Coder 两种交付格式

### 12.2 底层层验收
- 任务状态可创建、更新、恢复
- Resume State 可独立恢复主要上下文
- 关键写入具备安全写入能力
- 风险分级行为一致且有留痕
- 新旧结构兼容读取可工作

---

## 13. 最终结论

本方案的核心不是“做一个更会说的 Agent”，而是建立一个：
- 任务显式
- 事实优先
- 可恢复
- 可审计
- 可逐步迁移
的最小运行闭环。

本轮建议先做 Skill 和项目级状态底座，后做更深的底层分层与治理。
